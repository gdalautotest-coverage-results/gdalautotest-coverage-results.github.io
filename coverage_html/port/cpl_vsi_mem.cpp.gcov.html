      60             : /*
      61             : ** Notes on Multithreading:
      62             : **
      63             : ** VSIMemFilesystemHandler: This class maintains a mutex to protect
      64             : ** access and update of the oFileList array which has all the "files" in
      65             : ** the memory filesystem area.  It is expected that multiple threads would
      66             : ** want to create and read different files at the same time and so might
      67             : ** collide access oFileList without the mutex.
      68             : **
      69             : ** VSIMemFile: In theory we could allow different threads to update the
      70             : ** the same memory file, but for simplicity we restrict to single writer,
      71             : ** multiple reader as an expectation on the application code (not enforced
      72             : ** here), which means we don't need to do any protection of this class.
      73             : **
      74             : ** VSIMemHandle: This is essentially a "current location" representing
      75             : ** on accessor to a file, and is inherently intended only to be used in
      76             : ** a single thread.
      77             : **
      78             : ** In General:
      79             : **
      80             : ** Multiple threads accessing the memory filesystem are ok as long as
      81             : **  1) A given VSIMemHandle (i.e. FILE * at app level) isn't used by multiple
      82             : **     threads at once.
      83             : **  2) A given memory file isn't accessed by more than one thread unless
      84             : **     all threads are just reading.
      85             : */
